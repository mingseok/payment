# 결제 시스템

결제가 "되게" 만드는 데서 끝나지 않고, 결제 과정에서 생길 수 있는 실패, 중복, 정합성 문제에 대해 정리합니다.

구체적인 설계 방향은 Issues에 정리되어 있습니다. (링크 통해 설계 내용 확인)

<br/>


## 설계 방향

- [외부 결제 API 호출과 DB 저장을 한 트랜잭션으로 묶지 않는다](https://github.com/mingseok/payment/issues/10)

- [타임아웃을 결제 실패로 처리하지 않는다](https://github.com/mingseok/payment/issues/11)
- [승인 후 저장이 실패했을 때 결제 되돌리기](https://github.com/mingseok/payment/issues/12)
- [같은 주문 두 번 결제되지 않게 막는다](https://github.com/mingseok/payment/issues/13)
- [토스 응답을 그대로 믿지 않고 다시 검증한다](https://github.com/mingseok/payment/issues/14)

<br/>

## 그 외 신경 쓴 점

클라이언트 입력 검증
- [x] 주문 금액은 클라이언트가 아니라 서버가 상품 가격을 합산해 계산한다
- [x] 존재하지 않는 상품이 섞인 주문은 생성을 거부한다
- [x] 주문번호는 서버가 발급해 클라이언트가 정하지 못하게 한다

null과 빈 값 처리
- [x] 토스 응답에 결제 날짜가 없어도 현재 시각으로 채워 저장이 막히지 않게 한다
- [x] 주문 항목이 비어 있어도 기본 이름을 써서 빈 목록에서 터지지 않게 한다

주문 상태 변경 검사
- [x] 이미 결제된 주문을 다시 결제하는 식으로, 상태가 잘못된 순서로 바뀌는 것을 막는다
